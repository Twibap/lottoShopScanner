# 가까운 복권판매점 랭킹 앱 — MVP 제품 요구사항

## 1. 제품 개요

사용자의 현재 위치 또는 검색한 지역을 기준으로 가까운 동행복권 판매점을 찾고, 거리와 공개된 당첨 이력을 서로 혼동하지 않도록 구분해 비교할 수 있는 Android/iOS 앱을 만든다.

### 제품 한 줄 설명

> 내 주변 복권판매점의 위치와 공개 당첨 이력을 지도에서 빠르게 비교하는 앱

### MVP의 핵심 가치

- 지도와 목록에서 가까운 판매점을 빠르게 찾는다.
- 1등·2등 당첨 횟수와 최근 당첨 정보를 같은 기준으로 비교한다.
- 순위 산정 기준과 데이터 갱신 시점을 투명하게 보여준다.
- 선택한 판매점까지 외부 지도 앱으로 길찾기를 시작한다.

## 2. 전제와 제약

- 지원 대상은 대한민국 동행복권 판매점이다.
- 앱은 Android와 iOS를 동시에 지원한다.
- 기존 저장소의 `shops`, `winning_events`, `shop_statistics` 데이터를 활용한다.
- 판매점별 실제 판매량을 확보하지 못하면 당첨 이력은 당첨 확률을 의미하지 않는다.
- 사용자 위치는 주변 검색에만 사용하며, MVP에서는 서버에 위치 이력을 저장하지 않는다.
- 위치 권한을 거부해도 주소·지역 검색으로 모든 핵심 기능을 사용할 수 있어야 한다.
- MVP는 회원가입, 광고 식별자 수집, 맞춤형 광고를 사용하지 않는다.
- 최근 검색어는 기기에만 저장하고 사용자가 삭제할 수 있어야 한다.

## 3. 대상 사용자와 주요 상황

### 주요 사용자

1. 현재 위치에서 가장 가까운 판매점을 찾는 사용자
2. 이동 가능한 범위 안에서 과거 1등·2등 당첨 이력이 많은 판매점을 비교하는 사용자
3. 특정 동네나 여행지의 판매점을 미리 검색하는 사용자

### 핵심 사용자 여정

1. 앱을 처음 실행한다.
2. 위치 권한을 허용하거나 지역을 직접 검색한다.
3. 지도와 하단 목록에서 주변 판매점을 확인한다.
4. `가까운 순`, `1등 당첨 순`, `2등 당첨 순`, `당첨금 순`, `최근 당첨 순` 중 하나를 선택한다.
5. 판매점 상세에서 주소, 거리, 당첨 통계, 데이터 기준일을 확인한다.
6. 외부 지도 앱으로 길찾기를 시작한다.

## 4. MVP 범위

### 포함

- 현재 위치 조회 및 권한 처리
- 주소·행정구역·장소명 검색
- 지도 마커와 주변 판매점 목록
- 검색 반경 선택: 1km, 3km, 5km, 10km
- 정렬: 가까운 순, 1등 당첨 순, 2등 당첨 순, 당첨금 순, 최근 당첨 순
- 판매점 상세 정보
- 1등·2등 당첨 횟수와 최근 당첨 회차
- 자동/수동 당첨 방식 표시(원천 데이터가 제공하는 경우)
- 외부 지도 앱 길찾기
- 데이터 갱신 시점과 순위 기준 안내
- 네트워크·위치 권한·검색 결과 없음 상태

### 제외

- 회원가입과 로그인
- 사용자 리뷰, 별점, 사진 업로드
- 복권 구매 및 결제
- 번호 추천과 당첨 예측
- 푸시 알림
- 즐겨찾기와 기기 간 동기화
- 판매점 혼잡도 및 영업시간 보장
- 백그라운드 위치 추적

## 5. 정보 구조

```text
앱
├─ 탐색
│  ├─ 지역 검색
│  ├─ 지도
│  ├─ 주변 판매점 목록
│  └─ 반경·정렬 필터
├─ 판매점 상세
│  ├─ 기본 정보
│  ├─ 당첨 통계
│  ├─ 당첨 이력
│  └─ 길찾기
└─ 정보
   ├─ 랭킹 산정 기준
   ├─ 데이터 기준일
   └─ 위치·데이터 안내
```

MVP는 하단 탭을 두지 않고 `탐색 → 상세`의 얕은 구조로 구성한다. 랭킹 기준 안내는 탐색 화면의 정보 버튼과 상세 화면에서 진입한다.

## 6. 화면 명세

### S01. 시작 및 위치 권한

목적: 위치 사용의 이유를 설명하고 사용자가 탐색을 시작하게 한다.

- 위치 권한 요청 전에 앱 내부 설명을 먼저 보여준다.
- 주요 버튼: `내 위치로 찾기`
- 보조 버튼: `지역 직접 검색`
- 권한 거부 시 설정 이동을 강요하지 않고 지역 검색으로 연결한다.
- “위치는 주변 검색에만 사용하며 서버에 저장하지 않습니다” 문구를 노출한다.

### S02. 주변 판매점 탐색

목적: 현재 지도 영역의 판매점과 순위를 한눈에 비교한다.

- 상단: 검색창, 현재 위치 버튼
- 본문: 지도, 판매점 마커, 선택 마커
- 하단 시트: 결과 수, 정렬 칩, 반경 필터, 판매점 카드 목록
- 카드 정보: 순번, 상호명, 거리, 주소 요약, 선택한 정렬 기준의 통계(당첨 횟수·당첨금·최근 회차)
- 지도 이동 후 `이 지역 다시 검색` 버튼을 제공한다.
- 정렬이 가까운 순이 아닐 때는 순위의 기준을 제목에 명시한다.
- 목록과 마커 선택 상태를 동기화한다.

### S03. 지역 검색

목적: 위치 권한 없이도 원하는 지역을 탐색한다.

- 최근 검색어는 기기에만 저장한다.
- 검색 결과에서 장소 또는 행정구역을 선택하면 해당 좌표로 지도를 이동한다.
- 검색 결과가 없으면 주소를 더 간단히 입력하도록 안내한다.

### S04. 판매점 상세

목적: 선택한 판매점의 방문 판단에 필요한 근거를 제공한다.

- 상호명, 전체 주소, 전화번호(있는 경우), 현재 기준 거리
- 1등 횟수, 2등 횟수, 누적 당첨금, 총 당첨 회차 수, 최근 당첨 회차
- 전국 순위는 상세 화면에서만 보조 정보로 표시하고 `현재 반경 내 순위`와 명확히 구분한다.
- 당첨금 통계에는 누적 명목 당첨금, 현재 반경 내 당첨금 순위, 전국 당첨금 순위, 집계 시작·최종 회차를 표시한다.
- 당첨 이력 목록: 회차, 등수, 자동/수동, 추첨일
- 데이터 기준일과 출처
- 고정 주요 버튼: `길찾기`
- “과거 당첨 이력은 향후 당첨 확률을 보장하지 않습니다” 안내

### S05. 랭킹 기준 안내

목적: 숫자의 의미와 한계를 투명하게 설명한다.

- 각 정렬 기준의 정의
- 공동 순위 처리 방식
- 집계 범위 및 갱신 시점
- 판매량이 반영되지 않는다는 한계
- 판매점·당첨 결과는 `동행복권 공개 데이터 기반`, 지도·지오코딩은 `NAVER Maps`로 출처를 표시한다.
- 동행복권과 네이버가 제작·운영하거나 보증하는 앱이 아니라는 비공식 서비스 고지를 표시한다.
- 원천 데이터의 수정·지연·누락 가능성을 안내한다.
- `정보 오류 제보` 버튼은 전용 지원 이메일로 연결한다.

## 7. 랭킹 정의

MVP는 의미가 불명확한 단일 `종합 랭킹`을 만들지 않는다. 사용자가 선택한 지표별 순위를 제공한다.

| 기준 | 정의 | 동률 처리 |
|---|---|---|
| 가까운 순 | 기준 좌표와 판매점 좌표의 직선거리 | 거리 오름차순, 상호명 |
| 1등 당첨 순 | 수집 범위 내 1등 당첨 이벤트 수 | 2등 횟수, 최근 당첨 회차, 상호명 |
| 2등 당첨 순 | 수집 범위 내 2등 당첨 이벤트 수 | 1등 횟수, 최근 당첨 회차, 상호명 |
| 당첨금 순 | 수집 범위 내 해당 판매점의 1·2등 당첨금 합계 | 1등 당첨금 합계, 최근 당첨 회차, 상호명 |
| 최근 당첨 순 | 가장 최근 1·2등 당첨 회차 | 1등 횟수, 2등 횟수, 상호명 |

### 표시 원칙

- `명당`, `당첨 확률`, `추천 확률`처럼 확률을 암시하는 표현을 사용하지 않는다.
- 검색 반경 밖의 판매점은 결과와 순위 계산에서 제외한다.
- 화면의 순번은 현재 필터 결과 안에서의 순서다. 전국 순위가 존재하면 별도 라벨로 표시한다.
- 데이터가 없는 판매점은 `당첨 이력 없음`으로 표시하며 `0%`로 표시하지 않는다.
- 기존 DB의 전국 통계는 판매점 상세 화면에서만 보조 정보로 사용하고, 앱의 주변 결과 순번과 혼용하지 않는다.
- 전국 순위에는 항상 `전국`, 주변 결과 순번에는 `현재 반경 내` 라벨을 표시한다.
- 누적 당첨금은 각 회차의 명목 당첨금을 합산하며 물가를 보정하지 않는다. 판매점 매출이나 미래 당첨 확률을 의미하지 않는다.

## 8. 기능 요구사항

### 위치와 검색

- FR-01: 앱은 사용자 동의 후 현재 좌표를 조회해야 한다.
- FR-02: 위치 권한 거부·영구 거부·서비스 꺼짐을 서로 다른 상태로 안내해야 한다.
- FR-03: 사용자는 위치 권한 없이 지역 검색을 통해 탐색할 수 있어야 한다.
- FR-04: 사용자는 지도 중심점을 기준으로 결과를 다시 검색할 수 있어야 한다.

### 판매점 조회

- FR-05: 앱은 기준 좌표와 반경에 포함된 판매점만 조회해야 한다.
- FR-06: 각 결과에 거리와 당첨 통계의 데이터 기준일을 제공해야 한다.
- FR-07: 좌표가 없는 판매점은 지도 검색 결과에서 제외하고 데이터 품질 대상으로 기록해야 한다.
- FR-08: 지도 마커와 목록 항목의 선택 상태가 일치해야 한다.

### 정렬과 상세

- FR-09: 사용자는 다섯 가지 정렬 기준을 즉시 전환할 수 있어야 한다.
- FR-10: 정렬 변경 시 현재 검색 위치와 반경을 유지해야 한다.
- FR-11: 상세 화면은 판매점의 집계 값과 개별 당첨 이력을 함께 제공해야 한다.
- FR-12: 길찾기는 설치된 지도 앱 또는 플랫폼 기본 지도 선택 화면으로 연결해야 한다.
- FR-13: 전국 순위는 판매점 상세 화면에서만 보조 정보로 제공하고 탐색 목록에는 표시하지 않아야 한다. 상세 화면은 누적 당첨금과 함께 현재 반경 내 당첨금 순위 및 전국 당첨금 순위를 제공해야 한다.

### 데이터 신뢰성

- FR-14: 모든 통계 화면에 마지막 데이터 갱신 시점을 노출해야 한다.
- FR-15: 당첨 이력이 확률 예측이 아니라는 안내를 제공해야 한다.
- FR-16: 동일 판매점 식별은 이름·주소가 아니라 원천 `shop_id`를 기준으로 해야 한다.
- FR-17: 최신 회차 수집은 매주 토요일 21:00(KST)에 실행해야 한다.
- FR-18: 토요일 정기 수집이 실패하거나 최신 회차가 아직 게시되지 않은 경우 일요일 00:00, 06:00, 09:00, 12:00(KST)에 순서대로 재시도해야 한다.
- FR-19: 최신 회차 수집과 적재가 성공하면 해당 회차의 나머지 예약 재시도는 실행하지 않아야 한다.
- FR-20: 정보 오류 제보는 전용 지원 이메일로 접수하며 위치·주소, 폐점·상호, 당첨 이력, 기타 유형을 선택할 수 있어야 한다.
- FR-21: 제보 이메일에는 판매점 ID, 상호명, 앱 버전, 데이터 기준 회차만 자동 입력하고 사용자 위치, 기기 식별자, 진단 로그는 자동 첨부하지 않아야 한다.
- FR-22: 운영자는 제보를 3영업일 이내에 확인하고 원천 데이터와 대조한 뒤 수정하며 변경 이력을 기록해야 한다.

## 9. 비기능 요구사항

- 첫 결과 화면은 일반적인 LTE/5G 환경에서 목표 2초, 최대 4초 안에 표시한다.
- 지도 이동 중에는 요청을 연속 발생시키지 않도록 디바운스하거나 명시적 재검색을 사용한다.
- API 응답은 페이지네이션하며 한 번에 최대 100개 판매점을 반환한다.
- 모든 통신은 HTTPS를 사용한다.
- 서버 로그에 정밀 위치, 광고 식별자, 전화번호 같은 사용자 식별 정보를 저장하지 않는다.
- 분석 및 오류 로그에 사용자 좌표와 검색어를 포함하지 않는다.
- 개인정보처리방침은 공개 웹페이지로 제공하고 앱 설정과 Android/iOS 스토어 등록 정보에서 연결한다.
- 지도 등 외부 서비스로 전송되는 정보와 처리 목적을 개인정보처리방침에 명시한다.
- 출처 명칭이나 로고를 앱 이름 또는 앱 아이콘에 사용하지 않는다.
- NAVER Maps의 로고와 저작권 표시는 SDK 및 서비스 정책을 준수한다.
- 한국어를 기본 언어로 하고 글자 크기 확대와 스크린 리더 레이블을 지원한다.
- 최소 지원 버전은 구현 착수 시점의 Flutter 및 스토어 정책을 확인해 확정한다.

## 10. 권장 기술 구조

```text
Flutter 앱 (Android/iOS)
    ↓ HTTPS
조회 API
    ↓
PostgreSQL + 공간 검색 확장(PostGIS 권장)
    ↑
기존 Python 데이터 수집기
```

- 모바일: Flutter
- 상태 관리: Riverpod
- 라우팅: go_router
- API: FastAPI 또는 팀이 익숙한 동등한 REST 프레임워크
- DB: 기존 PostgreSQL에 PostGIS를 추가해 반경 검색과 거리 계산
- 지도·지오코딩: 네이버 Maps를 사용한다. Flutter 연동 플러그인의 유지보수 상태는 구현 착수 전에 검증한다.
- 지도 비용 정책: 초기 월 0원을 목표로 하며 5천 원에 예산 경고, 1만 원에 월간 사용 한도를 적용한다. 지오코딩 결과는 서버에 캐시한다.

## 11. 초기 API 계약

### 주변 판매점

`GET /v1/shops/nearby?lat={lat}&lng={lng}&radius_m={radius}&sort={sort}&limit={limit}&cursor={cursor}`

`sort`: `distance`, `first_wins`, `second_wins`, `total_prize`, `recent_win`

필수 응답 필드:

- `shop_id`, `name`, `address`, `latitude`, `longitude`
- `distance_m`
- `first_count`, `second_count`, `total_prize`, `winning_draw_count`, `last_winning_draw`
- `result_rank`, `updated_at`
- `next_cursor`

### 판매점 상세

`GET /v1/shops/{shop_id}`

### 판매점 당첨 이력

`GET /v1/shops/{shop_id}/winning-events?limit={limit}&cursor={cursor}`

## 12. 핵심 상태와 오류 처리

- 최초 로딩: 지도 스켈레톤과 결과 로딩 표시
- 위치 권한 거부: 지역 검색 CTA 제공
- 위치 서비스 꺼짐: 설정 안내와 지역 검색 CTA 제공
- 결과 없음: 반경 확대 또는 지도 이동 제안
- 오프라인: 재시도 버튼과 마지막 성공 결과가 있으면 오래된 데이터임을 표시
- API 오류: 기술 오류 코드 대신 사용자 행동이 가능한 메시지 제공
- 일부 통계 누락: 판매점 자체는 표시하고 해당 필드만 `정보 없음` 처리

## 14. MVP 완료 기준

- Android와 iOS 실제 기기에서 위치 권한 허용·거부 흐름이 동작한다.
- 현재 위치 또는 지역 검색으로 반경 내 판매점을 지도와 목록에서 볼 수 있다.
- 다섯 가지 정렬 결과가 API 정의와 일치한다.
- 판매점 상세에서 통계, 당첨 이력, 기준일을 확인할 수 있다.
- 길찾기가 플랫폼별 지도 앱으로 정상 연결된다.
- 판매량 미반영 및 확률 비보장 안내가 랭킹 설명과 상세에 노출된다.
- 주요 로딩·빈 결과·오프라인·서버 오류 상태가 처리된다.
- 위치와 랭킹 로직에 대한 자동 테스트 및 핵심 사용자 흐름 테스트가 통과한다.

## 16. 구현 전 확정할 결정 사항

- [확정] 지도·지오코딩은 네이버 Maps를 사용한다. 초기 목표 비용은 월 0원, 예산 경고는 5천 원, 월간 사용 한도는 1만 원으로 한다.
- [확정] 데이터는 1회부터 최신 회차까지 수집한다. 최신 회차는 매주 토요일 21:00(KST)에 갱신하고, 실패 또는 미게시 시 일요일 00:00, 06:00, 09:00, 12:00(KST)에 재시도한다. 성공 이후의 예약 재시도는 건너뛴다.
- [확정] 현재 반경 내 순위는 탐색 화면의 주요 정보로 표시하고, 전국 순위는 판매점 상세 화면에서만 보조 정보로 표시한다. 두 순위에는 각각 모집단을 나타내는 라벨을 붙인다.
- [확정] `total_prize_rank`는 판매점 상세 화면에서 `전국 당첨금 순위`로 제공한다. 누적 당첨금은 물가를 보정하지 않은 명목 당첨금 합계이며 판매점 매출이나 미래 당첨 확률과 무관함을 안내한다.
- [확정] 회원가입·광고 식별자·맞춤형 광고를 사용하지 않는다. 현재 위치는 주변 검색에만 사용하고 위치 이력은 저장하지 않는다. 최근 검색어는 기기에만 저장하며 삭제 기능을 제공한다. 분석·오류 로그에는 좌표와 검색어를 남기지 않는다. 공개된 개인정보처리방침을 앱과 양쪽 스토어에서 연결하고 외부 서비스로 전송되는 정보를 명시한다.
- [확정] 판매점·당첨 결과는 `동행복권 공개 데이터 기반`, 지도·지오코딩은 `NAVER Maps`로 표시한다. 상세·랭킹 안내 화면에 출처와 최종 갱신 시각을 제공하고 정보 화면에서 원본 사이트를 연결한다. 비공식 서비스 고지와 원천 데이터의 수정·지연·누락 가능성을 안내하며 각 공급자의 표시 정책을 준수한다.
- [확정] 앱의 정보 화면과 판매점 상세에서 `정보 오류 제보` 버튼을 제공하고 전용 지원 이메일로 접수한다. 제보 유형은 위치·주소, 폐점·상호, 당첨 이력, 기타로 구분한다. 판매점 ID, 상호명, 앱 버전, 데이터 기준 회차만 자동 입력하며 사용자 위치, 기기 식별자, 진단 로그는 자동 첨부하지 않는다. 3영업일 이내 확인을 목표로 하고 원천 데이터 대조 후 수정 이력을 기록한다. 실제 이메일 주소는 스토어 출시 전에 확정한다.
